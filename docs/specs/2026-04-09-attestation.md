# Система аттестаций — Спецификация

**Дата:** 2026-04-09  
**Статус:** Готова к реализации  
**Приоритет:** Высокий (ближайшая аттестация)

---

## Краткое описание

Система аттестаций заменяет Яндекс.Формы и ручную обработку данных.  
Цепочка: **Заявка → Проверка требований → Предаттестация 1 → Предаттестация 2 → Аттестация → Результаты**  
Все данные хранятся в CRM и привязываются к карточке ученика.

---

## Проблема

- Данные об аттестациях теряются (нет единого хранилища)
- Ручная обработка Яндекс.Форм — двойная работа
- Нет истории: когда сдавал, на каком кю, какие были замечания
- Нет проверки нормативов (срок с последней аттестации, посещения, возраст)
- Оплата не отражается в финансах

---

## Участники

| Роль | Что делает |
|------|-----------|
| Родитель/Ученик | Подаёт заявку через личный кабинет |
| Тренер/Администратор | Проверяет заявки, проводит предаттестации, вносит результаты, фиксирует оплату |
| Приглашённый сенсей | Оценивает на основной аттестации (результат вносит тренер) |

---

## Временная шкала аттестационного цикла

```
Т-8 нед.   Открытие приёма заявок
Т-6 нед.   Предаттестация 1
Т-1..2 нед. Предаттестация 2 (минимум за неделю до аттестации)
День X      Семинар (2-3 дня) + Основная аттестация
```

## Данные модели (SQL)

### Таблица `attestation_events` — сессия аттестации (раз в полгода)

```sql
CREATE TABLE attestation_events (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,                         -- "Аттестация весна 2026"
  discipline TEXT NOT NULL                     -- 'aikido' | 'wushu' | 'both'
    CHECK (discipline IN ('aikido', 'wushu', 'both')),
  applications_open_at DATE,                   -- когда открываются заявки
  preatt1_date DATE,                           -- дата предаттестации 1
  preatt2_date DATE,                           -- дата предаттестации 2
  event_date DATE NOT NULL,                    -- дата основной аттестации
  status TEXT DEFAULT 'draft'
    CHECK (status IN ('draft', 'open', 'in_progress', 'completed')),
  notes TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

### Таблица `attestation_applications` — заявка ученика

```sql
CREATE TABLE attestation_applications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id UUID REFERENCES attestation_events(id) ON DELETE CASCADE,
  student_id UUID REFERENCES students(id) ON DELETE CASCADE,

  -- Данные заявки
  discipline TEXT NOT NULL CHECK (discipline IN ('aikido', 'wushu')),
  current_grade TEXT NOT NULL,                 -- текущий кю/степень (напр. "7 кю")
  target_grade TEXT NOT NULL,                  -- на какой сдаёт (напр. "6 кю")
  last_attestation_date DATE,                  -- дата предыдущей аттестации
  last_attestation_grade TEXT,                 -- на каком кю последний раз сдавал

  -- Проверка требований (авто + ручное)
  req_tenure_ok BOOLEAN,                       -- срок с последней аттестации OK?
  req_visits_ok BOOLEAN,                       -- минимум посещений OK?
  req_age_ok BOOLEAN,                          -- возраст OK?
  req_override_by TEXT,                        -- если тренер вручную допустил — его имя
  req_override_note TEXT,

  -- Оплата
  price NUMERIC(10,2),                         -- стоимость (автоматически из прайса)
  paid BOOLEAN DEFAULT false,
  paid_at DATE,
  payment_method TEXT CHECK (payment_method IN ('cash', 'card', 'transfer')),

  -- Предаттестация 1
  preatt1_status TEXT                          -- 'approved' | 'conditional' | 'rejected'
    CHECK (preatt1_status IN ('approved', 'conditional', 'rejected')),
  preatt1_notes TEXT,
  preatt1_date DATE,
  preatt1_trainer TEXT,

  -- Предаттестация 2
  preatt2_status TEXT
    CHECK (preatt2_status IN ('approved', 'conditional', 'rejected')),
  preatt2_notes TEXT,
  preatt2_date DATE,
  preatt2_trainer TEXT,

  -- Итог основной аттестации
  result TEXT                                  -- 'passed' | 'passed_remarks' | 'failed'
    CHECK (result IN ('passed', 'passed_remarks', 'failed')),
  result_grade TEXT,                           -- присвоенный кю/степень
  sensei_notes TEXT,                           -- замечания сенсея
  result_date DATE,

  -- Служебное
  status TEXT DEFAULT 'pending'
    CHECK (status IN ('pending', 'approved', 'rejected', 'completed')),
  submitted_at TIMESTAMPTZ DEFAULT now(),
  created_at TIMESTAMPTZ DEFAULT now(),

  UNIQUE(event_id, student_id, discipline)
);

ALTER TABLE attestation_applications ENABLE ROW LEVEL SECURITY;
CREATE POLICY "allow_all" ON attestation_applications FOR ALL USING (true) WITH CHECK (true);
ALTER TABLE attestation_events ENABLE ROW LEVEL SECURITY;
CREATE POLICY "allow_all" ON attestation_events FOR ALL USING (true) WITH CHECK (true);
```

## Функциональные требования

### P0 — Обязательно к ближайшей аттестации

#### CRM — Раздел "Аттестации"
- [ ] Создать аттестационное мероприятие (название, дисциплина, даты)
- [ ] Список заявок: ученик / кю / дисциплина / проверка требований / предатт.1 / предатт.2 / оплата / результат
- [ ] Открыть/закрыть приём заявок (статус события: draft → open → in_progress → completed)
- [ ] Фильтр по дисциплине (айкидо / ушу)

#### CRM — Работа с заявкой
- [ ] Просмотр заявки с предупреждениями о непройденных требованиях
- [ ] Вручную допустить при несоответствии (поле `req_override_note`)
- [ ] Заполнить предаттестацию 1: статус (допущен / условно / не допущен) + замечания текстом
- [ ] Заполнить предаттестацию 2: аналогично
- [ ] Отметить оплату: сумма / наличные / карта + автоматически создаётся запись в `payments`
- [ ] Внести результат аттестации: прошёл / с замечаниями / не сдал + кю + текст сенсея
- [ ] После внесения результата — автоматически обновляется `certifications` в карточке ученика

#### Кабинет родителя/ученика
- [ ] Кнопка "Подать заявку на аттестацию" (если есть открытое мероприятие)
- [ ] Форма заявки: дисциплина, текущий кю, дата последней аттестации (можно из истории)
- [ ] Если данные есть в CRM — подтянуть автоматически, дать возможность поправить
- [ ] Предупреждение если не проходит по нормативам ("Требуется минимум 6 месяцев с последней аттестации — у вас 4. Заявка будет на рассмотрении тренера")
- [ ] Статус заявки виден в кабинете: ожидает / одобрена / результаты предатт. / оплатить / итог

#### Уведомления (Telegram → родителю)
- [ ] Заявка принята/отклонена (+ причина)
- [ ] Результат предаттестации + замечания
- [ ] Напоминание об оплате (после одобрения заявки)
- [ ] Результат аттестации (кю + оценка + замечания сенсея)

#### Карточка ученика
- [ ] Блок "История аттестаций": список всех аттестаций (дата / кю / оценка / замечания)
- [ ] Текущий кю/степень обновляется автоматически при успешной аттестации

### P1 — После ближайшей аттестации

- [ ] Автоматическая проверка нормативов по таблице кю (когда таблица будет предоставлена)
- [ ] Экспорт списка участников в CSV/Excel
- [ ] Статистика аттестации: сдали / условно / не сдали / не пришли

### P2 — Будущее

- [ ] Семинар как отдельный подтип мероприятия с регистрацией
- [ ] Отметка посещения семинара в истории ученика
- [ ] Обязательное условие: посещение семинара перед аттестацией по уровню
