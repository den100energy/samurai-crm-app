## Общая концепция

Добавляем в Samurai CRM три новые системы поверх уже готовой базы:

1. **Воронка текущих клиентов** — видим кто "в зоне риска" до того как они ушли
2. **Воронка спящих/ушедших** — возвращаем тех кто перестал ходить
3. **Монетизация** — апсейлы, кроссейлы, даунсейлы в правильный момент

Почти всё строится на данных которые уже есть: посещаемость, абонементы, срезы прогресса.

## Фаза 4A — Фундамент: статусы и алерты ⬜

**Цель:** Видеть "температуру" каждого ученика в реальном времени.

### Что делаем

**1. Статус ученика в базе данных**

Добавляем поле `crm_status` в таблицу `students`:

| Статус | Описание |
|--------|----------|
| `onboarding` | Первые 30 дней с момента зачисления |
| `active` | Регулярно посещает, всё хорошо |
| `at_risk` | Тревожный сигнал — нужна реакция |
| `established` | 6+ месяцев, лояльный |
| `champion` | 12+ месяцев + реферал |
| `paused` | Согласованная пауза (болезнь, каникулы) |
| `lapsed` | Прекратил платить без паузы |
| `churned` | Подтверждённый уход |

**2. Cron: еженедельный пересчёт статусов**

Каждое воскресенье ночью считаем "балл риска" для каждого ученика:

| Сигнал | Баллы |
|--------|-------|
| Не был 7–14 дней (при активном абонементе) | +1 |
| Не был 14–21 день | +3 |
| Посещаемость <50% за месяц | +2 |
| Абонемент просрочен >3 дней | +3 |
| Задача тренера по этому ученику не закрыта 7+ дней | +1 |

Если сумма ≥ 5 → автоматически ставим `at_risk`

**3. Telegram-алерт тренеру при at_risk**

"⚠️ Дмитрий Петров не был на тренировке 16 дней. Абонемент истекает через 4 дня. Рекомендую связаться."

**4. Раздел "Воронка текущих клиентов" в дашборде**

Колонки: `onboarding` | `active` | `at_risk` | `established` | `champion`

Кликабельно → список учеников в каждом статусе.

### Что нужно технически — КОНКРЕТНЫЕ ШАГИ

**Шаг 1 — SQL-миграция (выполнить в Supabase Dashboard → SQL Editor)**
```sql
ALTER TABLE students
  ADD COLUMN IF NOT EXISTS crm_status TEXT DEFAULT 'active',
  ADD COLUMN IF NOT EXISTS risk_score INTEGER DEFAULT 0,
  ADD COLUMN IF NOT EXISTS crm_status_changed_at TIMESTAMPTZ,
  ADD COLUMN IF NOT EXISTS crm_note TEXT;
```
Значения crm_status: `onboarding` | `active` | `at_risk` | `established` | `champion` | `paused` | `lapsed` | `churned`

**Шаг 2 — API `/api/cron/update-crm-status/route.ts`**
Логика (каждое воскресенье 08:00):
- Берём всех студентов со status='active'
- Для каждого: считаем дни с последнего визита + посещаемость за 30 дней + статус абонемента
- Если enrollment_date < 30 дней назад → `onboarding`
- Если enrollment_date > 180 дней и attendance_rate > 70% → `established`
- Иначе → `active`
- Затем считаем risk_score по таблице баллов
- Если risk_score ≥ 5 → crm_status = `at_risk` + Telegram-алерт тренеру
- Сохраняем crm_status, risk_score, crm_status_changed_at

**Шаг 3 — Виджет "CRM воронка" на дашборде (page.tsx)**
- Новый стейт `crmFunnel: Record<crm_status, {id,name}[]>`
- Горизонтальный ряд карточек: onboarding | active | at_risk | established | champion
- at_risk — красная подсветка, кликабельно → модалка со списком

**Шаг 4 — vercel.json**
```json
{ "path": "/api/cron/update-crm-status", "schedule": "0 8 * * 0" }
```

**Шаг 5 — Telegram-алерт при at_risk**
Отправляется владельцу + тренеру студента:
`⚠️ [Имя] не был на тренировке N дней. Абонемент истекает через X дней. Балл риска: Y.`

## Фаза 4B — Воронка текущих клиентов ✅ ГОТОВО (2026-03-25)

**Цель:** Системно прогревать каждого ученика на всех этапах жизненного цикла.

### Онбординг (первые 30 дней)

**Автоматические touchpoints:**

| Событие | Канал | Что отправляем |
|---------|-------|---------------|
| Зачисление | Telegram (родителю) | Приветствие, что взять на тренировку, расписание |
| После первого занятия | Задача тренеру | "Написать ученику: как прошло первое занятие?" |
| День 7 | Задача администратору | "Позвонить/написать: удобно ли расписание?" |
| День 30 | Telegram (родителю) | Итог месяца: N занятий, что освоил, следующий этап |

**Триггер тревоги при онбординге:**
- 0 занятий за первые 14 дней → немедленная задача тренеру

### Активный ученик

**Автоматические milestones:**

| Триггер | Действие |
|---------|---------|
| 30-е занятие | Значок в кабинете + Telegram родителю |
| 60-е занятие | Значок |
| 100-е занятие | Значок + поздравление от школы |
| 3 месяца (≥70% посещаемость) | Задача тренеру: предложить годовой абонемент |
| 6 месяцев | Задача тренеру: "Progress conversation" |
| 12 месяцев | Годовщина — личное сообщение + бонус |
| День рождения | Telegram от тренера |

**Работа со статусом at_risk (при срабатывании алерта):**

Тренер видит в задаче:
- Имя ученика + последний визит
- Балл риска и почему сработало
- Быстрые кнопки действий: "Написал" / "Поставить на паузу" / "Ушёл"
- Поле для заметки о причине

### Что нужно технически
- Таблица `lifecycle_tasks` (или использовать существующую систему тикетов)
- API `/api/cron/lifecycle-touchpoints` — ежедневная генерация задач по триггерам
- Значки milestones в кабинете ученика
- Дополнить карточку ученика: статус + история изменений статуса

## Фаза 4C — Воронка спящих и ушедших клиентов ✅ ГОТОВО (2026-03-25)

**Цель:** Системно работать с теми кто перестал ходить — вместо ручного "пытаемся связаться".

### Сегменты (автоматически определяются)

| Сегмент | Условие | Приоритет |
|---------|---------|-----------|
| Ранний сигнал | Не был 10–21 день + абонемент активен | Критический |
| Спящий | Не был 21–90 дней ИЛИ абонемент истёк 1–60 дней | Высокий |
| Ушедший | Не был 90–180 дней, нет абонемента | Средний |
| Потерянный | 180+ дней | Только новости |

### Раздел "Возврат клиентов" в интерфейсе

Отдельная страница `/leads` расширяется вкладкой "Спящие":

- Таблица с сегментами (ранний / спящий / ушедший)
- Для каждого ученика: последний визит, причина (если знаем), рекомендуемое действие
- Кнопки: "Отправить сообщение" / "Поставить задачу" / "Предложить паузу" / "Подтвердить уход"

### Сценарии возврата (шаблоны + задачи тренеру)

**Ранний сигнал (10–21 день):**
Задача тренеру — личное сообщение (не шаблон). Текст-подсказка в интерфейсе.

**Спящий (21–90 дней):**
- Шаг 1: Личное сообщение с "паузой или возвратом?"
- Шаг 2 (через 14 дней без ответа): Конкретное предложение с дедлайном
- Шаг 3 (через 14 дней): Мягкое завершение + крюк возврата

**Ушедший (90–180 дней):**
- Одна кампания. Самое сильное предложение. Больше не повторять.

### Даунсейл-воронка (вместо просто "ухода")

Когда ученик сигнализирует об уходе — тренер видит в интерфейсе предложения:

1. **Заморозка** — "Предложить заморозку на 1–2 мес (бесплатно)"
2. **Сокращённый план** — "Перевести на 4 занятия/мес (режим поддержки)"
3. **Выход с крюком** — "Оформить уход + пообещать сохранить пояс при возврате в 6 мес"

CRM фиксирует какой вариант предложен и чем закончилось.

### Что нужно технически
- Cron: ежедневное обновление сегментов спящих/ушедших
- Страница `/students/win-back` (или вкладка в `/leads`)
- Шаблоны сообщений в интерфейсе (тренер редактирует и отправляет через Telegram-бот)
- Поле `churn_reason` + `downsell_offered` + `downsell_result` в `students`
- Telegram-уведомление: "У вас 3 спящих ученика требуют внимания"

## Фаза 4D — Апсейлы и кроссейлы ⬜

**Цель:** Зарабатывать больше с тех кто уже с нами.

### Апсейл: "У потолка абонемента"

Когда ученик посетил N-1 из N занятий по плану — в кабинете родителя появляется баннер:
"Артём посетил 7 из 8 занятий этого месяца. Безлимитный абонемент обошёлся бы дешевле."

Плюс задача администратору.

### Апсейл: Годовой абонемент

Через 3 месяца регулярного посещения (посещаемость >70%) — задача тренеру:
"Предложить годовой абонемент [Имя] — скидка 15% от месячной цены × 12."

### Апсейл: Индивидуальные занятия

Триггеры:
- Средний балл тренера ≥70/100 И нет ни одного индивидуального занятия
- Ученик готовится к аттестации (за 3–4 недели)
- Ученик готовится к соревнованию

### Кроссейл: Семейный абонемент

На карточке ученика добавляем поле "Братья/сёстры". Через 60 дней после зачисления — если поле заполнено — задача:
"Предложить семейный абонемент [семья Петровых]."

### Кроссейл: Лагерь/мероприятие

За 6 недель до каникул — автоматическое уведомление всем активным ученикам о записи на лагерь (если лагерь создан в системе).

### Что нужно технически
- Поле "родственники" / "братья-сёстры" в карточке ученика
- Баннер в кабинете родителя: "ученик у потолка плана"
- Cron: ежемесячный пересчёт триггеров апсейла → задачи тренерам
- Модуль регистрации на лагеря/события (если ещё нет)

## Фаза 4E — Лояльность и "Паспорт прогресса" ⬜

**Цель:** Сделать так, чтобы уйти = бросить свою историю.

### Значки посещаемости (stickers/achievements)

В кабинете ученика добавляем раздел "Достижения":
- 🔥 Стрик: 30 занятий без пропуска
- 💯 Стрик: 3 месяца с посещаемостью 90%+
- 🥋 Каждый полученный пояс
- 🏕️ Каждый лагерь
- 🏆 Каждое соревнование

### "Паспорт прогресса" — хронология в кабинете

Единая вертикальная лента событий в кабинете ученика (снизу вверх — от начала до сегодня):

```
▲ Март 2026
  ● Срез прогресса "Рост — Март 2026"
  ● 100-е занятие
  ● Оранжевый пояс 🥋

▲ Январь 2026
  ● Зимний лагерь "Самурай"
  ● Соревнование — 2 место

▲ Октябрь 2025
  ● Первый срез прогресса

▲ Сентябрь 2025
  ● Начало пути в Школе Самурай
```

Большинство данных уже есть — нужно только собрать в единый UI.

### Реферальный учёт

На карточке лида добавить поле "Кто привёл". При конвертации — привязать к карточке ученика.
Статистика на дашборде: "15% новых учеников пришли по рекомендации."

### Что нужно технически
- Таблица `student_achievements` (или расширить существующую)
- Компонент "Лента событий" в кабинете ученика/родителя
- Поле `referred_by` в таблице лидов/учеников
- Cron: автоматическое выдавание значков по триггерам
